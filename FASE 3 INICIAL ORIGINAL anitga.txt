fASE 3 EDITADA (3):
3. Despertar da Criogenia (Warmups e GC Tático)
3.1 O Diagnóstico Sênior e Fluxo Atual
Em 
PoolService.ts
 a inicialização (PoolService.get) é majestosa, porém lazily executada.
Um tenant inativo por +5 minutos vê a API do Cascata responder o primeiro request da manhã em ~300ms a 600ms devido a: "TCP Handshake API -> DB, TLS Negotiation Auth DB" até a resposta.
O lixeiro (
reapZombies
) executa uma poda agressiva da memória do Node.js baseando-se unicamente em um limite de inatividade engessado: 5 Minutos para todos.
3.2 O Plano Cirúrgico (Inteligência Artificial Algorítmica nos Pools)
A arquitetura "Cold Start" precisa ser dilacerada.

Arquivos Alvo a Sofrerem Metamorfose:

backend/services/PoolService.ts
:
Implementação de Tiers de Ociosidade Algoritmica (IDLE_THRESHOLD dinâmico). Se um projeto for assinado/identificado pelas chaves do banco de dados como nível "Enterprise", o Reaper não ceifa o Pool dele por 1 Hora. Planos grátis ou padrões permanecem restritos a 5m/10m.
Incorporação do método de Cache Paralelo. Ao acordarmos o cluster em 
server.ts
, instanciamos os Pools dos TOP 30 ou TOP X Projetos de modo asíncrono em plano de fundo sem sequer ter havido request para eles, tornando o acesso 0ms instantâneo logo no ar.
backend/server.ts
:
Chamamento proativo dessas lógicas no BOOT (System Bootup), aproveitando a ausência de concorrência massiva nesse ínfimo milissegundo em que o servidor levanta.
