A arquitetura de Key Groups funciona através de uma hierarquia onde o Key Group (o "Plano") são incoporadas as regras do jogo qunado se faala em tabelas(crud personalizados) ou uso de RPC.

 - como "Plano Gold", "Parceiros B2B" ou "Free Tier" - contendo a configuração JSON com crud_limits (exemplo: leitura ilimitada, escrita 100/s) e permissões de RPC, enquanto a API Key (o "Crachá")  está vinculada a um grupo através de um campo adicional novo, podendo ser movida entre grupos conforme a necessidade do administrador.
 
 A estratégia "Zero-DB" protege os Pools de conexão contra ataques através de validação em memória (Redis), onde as configurações dos grupos ficam cacheadas (config:group:gold -> { "crud": { "read": -1, "create": 50 } "rpc_whitelist": ["calcular_frete"] 10 }) e a validação de requisições verifica no Redis se a chave existe e qual seu grupo (apikey:sk_live_xyz -> { "group_id": "gold", "vencimento": "12/12/2032" }), rejeitando requisições inválidas ou vencidas antes de qualquer conexão com o Postgres ser aberta(Mas isso dependde das configurações, por exemplo se o adminissstrador na horaa da criação de tal regra ativar um "if"
