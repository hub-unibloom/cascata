Este documento define as regras de "Como escrever o código" para o Cascata,
garantindo que a migração para Go e a manutenção do React/TS sigam padrões
de segurança extrema, performance e manutenibilidade.

## 1. DIRETRIZES TOTAIS (TRANSVERSAL)
- SECURITY FIRST: Toda função que manipula dados sensíveis deve ser documentada com os riscos e proteções implementadas.
- ZERO MOCK: Não permitimos funções "placeholder". Toda implementação deve ser real ou retornar erro explícito de "não implementado".
- NOMENCLATURA SENSÍVEL: Nunca use nomes genéricos como 'password', 'key', 'token' para variáveis em memória. 
  * Use prefixos que descrevam o estado: `hashedPassword`, `encryptedVaultKey`, `rawInputSecret`.
- AUTO-EXPLICAÇÃO: O código deve ser legível. Comentários devem explicar o "PORQUÊ" (decisão técnica), enquanto o nome da função explica o "O QUE".

## 2. GO (BACKEND CORE) - PADRÕES DE IMPLEMENTAÇÃO

### 2.1 ESTRUTURA DE PACOTES E NOMENCLATURA
- Nomes de pacotes: Minúsculos, palavra única, curtos (ex: `auth`, `vault`, `driver`). Evite `utils` ou `helpers`.
- Funções e Variáveis Exportadas: PascalCase.
- Erros Exportados: Devem começar com `Err` (ex: `ErrProjectNotFound`).

### 2.2 ARQUITETURA EM CAMADAS (DRY / SOLID)
- HANDLERS: Apenas decodificação de request e codificação de response. Não contém lógica de negócio.
- SERVICES: Onde reside a lógica de negócio, orquestração de chamadas e validações complexas.
- REPOSITORIES: Abstração de acesso a dados (Postgres, Redis, Vault). Consultas puras.

### 2.3 TIPAGEM E RESPOSTAS DE API
- Toda resposta de API deve ter uma `struct` definida. **PROIBIDO** usar `map[string]any` ou `interface{}` para retornos JSON.
- Use `json:"field_name"` tags em todas as structs de transição de dados.

### 2.4 TRATAMENTO DE ERROS E CONTEXTO
- SEM PANIC: O uso de `panic` é proibido, exceto em inicializações críticas (main) onde o sistema não pode rodar sem o recurso.
- TRATAMENTO EXPLÍCITO: `if err != nil { return fmt.Errorf("contexto: %w", err) }` é obrigatório. Nunca silencie erros.
- CONTEXTO OBRIGATÓRIO: Toda função de I/O (DB, Redis, API, FileSystem) DEVE receber `ctx context.Context` como primeiro parâmetro.

### 2.5 LOGGING E MONITORAMENTO
- LOGS ESTRUTURADOS: Use `slog` (da standard library) ou similar. Nunca use `fmt.Println` ou `log.Print`.
- Campos de log devem ser consistentes (ex: `user_id`, `project_id`, `request_id`).
- TRACING E BREADCRUMBS: Toda requisição deve propagar um `RequestID` no contexto para rastreabilidade cross-service.

### 2.6 VALIDAÇÃO E SEGURANÇA (FAIL FAST)
- VALIDAÇÃO NA BORDA: Tudo que entra na API deve ser validado imediatamente (ex: `validator/v10` em Go). Se falhar, retorne `400 Bad Request` antes de qualquer lógica.
- CRIPTOGRAFIA: Funções de Vault ou Criptografia devem ser isoladas e auditáveis.

## 3. REACT / TYPESCRIPT (FRONTEND) - PADRÕES DE INTERFACE

### 3.1 COMPONENTES E LÓGICA
- APENAS FUNCTIONAL COMPONENTS: Uso estrito de Hooks (`useState`, `useEffect`, `useMemo`). Sem `Class Components`.
- CUSTOM HOOKS: Lógica complexa de componentes deve ser extraída para hooks customizados (ex: `useProjectVault`).

### 3.2 TIPAGEM ESTRITA (TYPE SAFETY)
- ZERO ANY: O uso de `any` é terminantemente proibido. Use `unknown` se o tipo for realmente incerto, mas prefira Generics.
- SEM "as unknown as Type": Evite type casting forçado. Se o tipo não bate, a interface/struct de origem está errada.

### 3.3 ESTADO E API
- ESTADO GLOBAL: Use apenas para dados que realmente atravessam toda a aplicação (Auth, Configurações Globais).
- CHAMADAS DE API: Nunca faça `fetch` ou `axios` direto no componente. Use uma camada de Service ou Hooks dedicados para abstrair a comunicação.

## 4. REGRAS DE ARQUIVOS (DOCKER / VPS)
- MANUTENÇÃO DE EXTENSÕES: Arquivos que terminam em `.txt` (como `Dockerfile.txt`, `.env.txt`) devem permanecer com essa nomenclatura no ambiente de desenvolvimento para compatibilidade com IDEs Web, conforme a Lei de Nomenclatura.
- REQUISITOS DE DOCKER: Todo novo serviço Go deve ser acompanhado de seu respectivo `Dockerfile.txt` otimizado para produção (distroless ou alpine).
