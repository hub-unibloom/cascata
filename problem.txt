## Bugs reais (comportamento errado)

**1. Padrão `err + FAIL=true` é dead code em toda a `preflight_checks` (linhas ~228, 246, 210)**

A função `err()` chama `exit 1` diretamente. Então em todo o bloco:

```bash
err "Espaço em disco insuficiente..."
FAIL=true   # ← nunca executa
```

O `FAIL=true` é código morto. A checagem final `if [ "$FAIL" = true ]` na linha 250 **jamais dispara** — o intent era acumular todos os erros pré-voo e exibir no final, mas na prática o script aborta no primeiro. Não quebra nada porque `err` sai mesmo, mas o comportamento é completamente diferente do que o código aparenta. É um bug de design que vai confundir quem tentar depurar ou modificar.

## Deprecated (vai gerar warnings ou quebrar em versões novas)

**2. `ChallengeResponseAuthentication` deprecated no OpenSSH 9.0+**

Linha 1482:
```
ChallengeResponseAuthentication yes
```

No OpenSSH 9.0 (2022), essa diretiva foi renomeada para `KbdInteractiveAuthentication`. O script **já tem `KbdInteractiveAuthentication yes` na linha 1483**, então a redundância é inócua em versões antigas, mas em Ubuntu 24.04 (OpenSSH 9.3+) a linha deprecated gera warning no `sshd -t`. O `sshd -t` na linha 1527 pode ou não falhar dependendo da versão — em algumas é só warning, em outras é erro fatal. Dado que você roda Ubuntu 24.04, isso vai aparecer. **Solução: remover a linha 1482 — a 1483 já cobre.**

## Code smells que podem virar bug

**3. `local DAEMON_JSON` declarado duas vezes na mesma função**

Linhas 1643 e 1659, dentro da mesma função, ambas fazem `local DAEMON_JSON="/etc/docker/daemon.json"`. Bash não reclamará, mas é uma armadilha — qualquer edição futura pode dessincronizar os dois valores. Shellcheck vai marcar isso como SC2155.

**4. Redirect redundante na linha 732**

```bash
if ls backend/migrations/*.sql.txt &>/dev/null 2>&1; then
```

`&>/dev/null` já redireciona stdout e stderr juntos. O `2>&1` depois é redundante. Não quebra nada, mas é ruído.

**5. `provision_vault_resources` definida como função aninhada dentro de `phase2_setup_and_start`**

Bash suporta isso, mas a função fica com escopo global após a primeira chamada. Se `phase2_setup_and_start` for chamada de novo (mesmo que improvável pelo state file), a função é redefinida. O padrão correto seria definir `provision_vault_resources` no nível top-level como todas as outras funções. Além de confuso pra ler.

**6. HARDENING_TOKEN não é idempotente entre re-runs**

Linha 1273 gera um token aleatório novo a cada execução. Se `PHASE3_DONE=false` e o script for re-executado, ele tenta `chattr -i` nos arquivos do token **novo** (que ainda não existem), mas os arquivos do token **anterior** ainda podem estar com `+i` ativo. Isso vai fazer a Fase 3 falhar ao tentar sobrescrever configs antigas congeladas. O `chattr -i` nas linhas 1316–1328 deveria ser sobre o token anterior (lido de algum manifest), não o novo.

**7. Sem error handling no pipeline do Vault token (linha 1104)**

```bash
BACKEND_TOKEN=$(vault_exec token create ... -format=json | python3 -c "...")
```

Se o `vault_exec` falhar ou retornar JSON malformado, o python3 levanta exceção e `BACKEND_TOKEN` fica vazio (ou pior, `set -e` aborta o script no meio da fase 2 sem mensagem amigável). Mínimo: adicionar `|| err "Falha ao gerar token do Vault"`.

## Minor (sem impacto em produção, mas vale corrigir)

`lsb_release -cs` na linha 290 é legacy em Ubuntu 22.04+ mas tem fallback para `$VERSION_CODENAME`. O comentário na linha 1633 tem typo ("atucha" → "atualiza"). `grep -oP` na linha 264 usa Perl regex — funciona em todas as distros alvo mas não é POSIX puro.

---

**Resumo executivo:** o script é sólido no geral — a estrutura de fases, a geração de credenciais, o hardening SSH e kernel estão bem implementados. Os dois problemas que eu priorizaria corrigir antes de produção são o `ChallengeResponseAuthentication` deprecated (vai quebrar silenciosamente em Ubuntu 24.04) e o HARDENING_TOKEN sem idempotência (vai travar re-runs na Fase 3).
