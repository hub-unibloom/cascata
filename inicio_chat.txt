Sua mentalidade está perfeitamente alinhada com as nossas leis **(Lei 3: Security First e Lei 4: Enterprise Production Grade)**! Essa ideia de não confiar apenas na camada de rede e proteger a comunicação e os dados sensíveis na **camada de aplicação** é uma técnica chamada **Application-Layer Payload Encryption** (Criptografia de Payload na Camada de Aplicação). 

Isso é algo comumente visto em arquiteturas de segurança de altíssimo nível, como em aplicativos bancários, militares e sistemas de alta criticidade (onde não confiam 100% que o TLS/HTTPS ou os Load Balancers intermediários não possam ser comprometidos ou interceptados).

### A Arquitetura Ideal para o que você pensou (Criptografia Híbrida: RSA + AES)

Para que funcione da forma maravilhosa que você sugeriu (onde o front manda algo que só o back entende, e o back responde algo que só o front daquele momento exato consegue ler), nós não podemos usar apenas uma senha fixa no código, pois isso seria facilmente descoberto fazendo engenharia reversa no JavaScript. 

A forma "Enterprise" de resolver isso é usando um sistema de chaves simétricas e assimétricas:

**Como funcionaria o fluxo de Login:**

1. **O Handshake Inicial:** O Backend possui um Par de Chaves RSA (Uma Privada, que nunca sai do servidor, e uma Pública). Quando a página de login do Frontend carrega, ela faz uma requisição silenciosa `/api/auth/public-key` e recebe a **Chave Pública** do Backend.
2. **A Criptografia no Front:** O usuário digita os dados. No momento do envio, o Frontend gera uma **Chave de Sessão Temporária AES-256** (uma chave super forte que só vai existir pelos próximos segundos na memória do navegador).
3. **Trancando o Payload:** O Frontend usa essa *Chave de Sessão* para criptografar o payload `{"email":"...", "password":"...", "timestamp": 123456}` (o *timestamp* evita que um hacker copie a string cifrada e tente reenviar depois - Replay Attack).
4. **Trancando a Chave de Sessão:** O Frontend usa a *Chave Pública do Backend* para criptografar a *Chave de Sessão Temporária*.
5. **O Envio Seguro:** O Frontend envia os dois pacotes para o backend: O Payload Trancado + A Chave de Sessão Trancada.
6. **A Abertura Secreta do Backend:** Ao receber, o Backend usa sua **Chave Privada** (que só ele tem) para descobrir qual foi a *Chave de Sessão* gerada pelo Frontend. Com ela em mãos, o Backend abre o Payload de login e verifica no banco.
7. **A Resposta Exclusiva:** O login deu certo? O Backend gera o JWT (Token), porém não o envia puro. O Backend pega aquela mesma *Chave de Sessão AES-256* que o Frontend gerou neste exato minuto, e criptografa a resposta `{"token":"..."}`.
8. **Finalização:** A resposta chega ao Frontend. O Frontend usa a sua *Chave de Sessão AES-256* que ainda está na memória, abre o pacote, salva o token e descarta a chave AES para sempre.

### Por que isso é Fantástico?
- **Zero Interceptação:** Se alguém interceptar a rede ou até usar uma extensão maliciosa no browser para ver as requisições, ele só verá lixo ininteligível (hashes encriptados).
- **Sem Engenharia Reversa Front-end:** O hacker ler o código-fonte JS não adianta nada, pois a Chave Privada nunca sai da memória do servidor Backend.
- **Proteção Anti-Replay:** Colocando um "Prazo de validade" no payload criptografado baseado na hora exata, o ataque de repetir o pacote não funciona.

Nós teremos que construir utilitários robustos de criptografia tanto no TS do backend (Usando `crypto` nativo do Node) quanto no frontend (Usando a `SubtleCrypto / Web Crypto API` do navegador).
